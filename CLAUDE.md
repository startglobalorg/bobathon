# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Apartner — a "Tinder for apartments" web app for the Zürich rental market. Mobile-first Next.js 14 (App Router) + TypeScript + Tailwind, with Prisma + SQLite for persistence and Anthropic SDK for cover-letter generation. The MVP is being built across parallel "lanes" (B/C/D) writing features against a shared schema; this repo is the foundation they all import from.

## Commands

```bash
# dev / build
npm run dev          # next dev on port 3006 (NOT 3000)
npm run build        # next build (output: 'standalone')
npm run start        # next start on 3006
npm run typecheck    # tsc --noEmit, strict mode
npm run lint         # next lint

# database (Prisma 6 + SQLite)
npm run db:migrate   # prisma migrate dev — creates a new migration in prisma/migrations/
npm run db:seed      # tsx prisma/seed.ts — wipes listings/applications, reseeds from data/listings.json, ensures profile id=1
npm run db:studio    # prisma studio
npm run db:reset     # prisma migrate reset --force — drops dev.db and re-runs migrations + seed
```

There are no tests in this repo yet.

## Architecture

### Port 3006 everywhere
Wired into `package.json` scripts, `Dockerfile` (`ENV PORT=3006`, `EXPOSE 3006`), and `docker-compose.yml` (`3006:3006`). Don't change one without changing all three.

### Prisma + SQLite — and the Prisma 6 pin
`prisma` and `@prisma/client` are pinned to `^6` **intentionally**. Prisma 7 dropped the inline `url = env("DATABASE_URL")` syntax in `schema.prisma` and requires a separate `prisma.config.ts`; the rest of the build plan assumes the v6 schema layout. Don't `npm i prisma@latest` unless you also migrate the datasource config.

The schema (`prisma/schema.prisma`) is the source of truth for all feature lanes:

- `Profile` (1:1 with `Preferences`, 1:N with `PreviousLandlord`, `Application`)
- `Listing` — string `id` (e.g. `zh-001`), with two SQLite-shaped fields: `gallery` is a **JSON-stringified array** of URL strings, and `Preferences.neighborhoods` is a **comma-separated string** (SQLite has no array type — handle (de)serialization at the boundary)
- `Application` — `@@unique([profileId, listingId])` enforces "one swipe per listing per user"

`lib/prisma.ts` exports a `PrismaClient` singleton stashed on `globalThis` to survive Next.js hot-reload. Always import it as `@/lib/prisma` (path alias from `tsconfig.json`); never `new PrismaClient()` in app code.

### Hardcoded `profileId = 1`
There is no auth seam in the MVP. The seed script ensures a profile with `id: 1` exists with empty defaults; all feature code reads/writes against that single profile. Don't add login flows unless the plan changes.

### Two listing data sources — don't conflate them
- `data/listings.json` — **seed input.** 3 placeholder Zürich listings shaped exactly to the Prisma `Listing` model. The `db:seed` script reads this. The team will swap this file for a hand-crafted 10-entry version later; the seed pipeline must keep working against whatever's there.
- `mock-data/listings.json` — **50 listings in Homegate-style nested shape** (`address.geoCoordinates.latitude`, `prices.rent.gross`, `localization.de.text.title`, etc.). Generated by `mock-data/generate_listings.py` for a different purpose (probably UI prototyping / scraper-replacement testing). It is **not** the seed input — the shapes don't match.

### Docker persistence model
The deploy target is Hetzner + Portainer + Pangolin. SQLite is a single file, so without volumes every redeploy wipes the DB. Two non-obvious wires that must stay aligned:

1. **`docker-compose.yml`** mounts named volumes `apartner-data:/app/data` and `apartner-uploads:/app/uploads`, and overrides `DATABASE_URL=file:/app/data/dev.db` (not `./dev.db`). Locally, `.env`'s `DATABASE_URL=file:./dev.db` wins.
2. **`Dockerfile`** must `mkdir -p /app/data /app/uploads && chown -R nextjs:nodejs ...` **before** the `USER nextjs` switch. Otherwise volumes mount as root-owned and the app can't write.
3. **`docker-entrypoint.sh`** runs `npx prisma migrate deploy` then `exec node server.js`. The runner stage of the Dockerfile copies `node_modules/prisma`, `node_modules/@prisma`, and `prisma/` into the image specifically so the entrypoint can do this — Next.js standalone tracing doesn't pull the CLI.

`prisma migrate dev` is dev-only; production never runs it. Migrations get committed (`prisma/migrations/`); `dev.db` is gitignored.

### Deploy: no CI, manual via Portainer
There used to be a `.github/workflows/build-and-push.yml` that pushed to GHCR; it was removed because the `GHCR_PAT` secret wasn't configured. Image is built locally and redeployed manually through Portainer. The compose image tag is `ghcr.io/startglobalorg/bobathon:latest`.

## Non-obvious conventions

- **No `any` types.** TS is strict and the build plan calls this out explicitly. Use `unknown` + narrowing.
- **`output: 'standalone'`** is on in `next.config.js` — required for the slim Docker runner stage.
- **`shadcn/ui` is initialized** (see `components.json`) but no components have been added yet; use `npx shadcn@latest add <component>` and they land in `components/ui/`.
- **Anthropic SDK is installed but not wired up.** When you wire it in, expect cover-letter generation flows triggered by `Application.status` transitions.
